## Context

The `add-schedule-management` change shipped `dump_controller_snapshot` with the explicit non-goal *"sensors, weather stations, gateways"*. Subsequent live verification against a 22-zone Pro-HC controller surfaced that **even within the original scope**, the snapshot is missing critical data:

- For STANDARD-mode controllers (this controller, and most accounts per CLAUDE.md), the watering schedule lives on `StandardProgram` records. The snapshot only captures program **id/name/zone-IDs** via `getPrograms`. The actual schedule (start times, days run, periodicity, per-zone run minutes, monthly adjustments, ignoreRainSensor, validFrom/validTo, conditional watering adjustments) requires `getStandardProgram` per program — not currently called.
- `Controller.location` (latitude, longitude, address) — required for Virtual Solar Sync to know where to look up weather. Restoring without it breaks Solar Sync.
- `Controller.settings.timeZone` — schedule wall-clock interpretation depends on this.
- `Controller.masterZone` + `Controller.settings.zones.{interZoneDelay, masterZone}` — pump/valve sequencing.
- `Zone.masterValve` (Int) — per-zone override (-1 = follow global, 0 = always disabled, else specific zone number).
- `Controller.expanders` — for Pro-HC accounts, zones 13-24 are usually on an expander. Restoration without expander config means those zones can't be addressed.
- `Controller.hardware.modules` — installed firmware modules.
- `Controller.runTimeGroups` — the named "30 minutes" / "40 minutes" groups that StandardProgram references via `runTimeGroup.id`. Capturing these alongside the programs makes the relationships restorable.
- `Controller.controllerNotes` and per-zone `Zone.zoneNotes` — user-written context that should survive restore.
- `ProgramStartTime.wateringDays` — currently unselected by the existing query, returning `null`.

The snapshot format is mutable (decision: AI is the sole consumer at restore time), so we can reshape the envelope freely. We don't need backward-compatible migration.

For the **write path**, the schema audit confirms every new field has a corresponding mutation:

| Field | Mutation |
|---|---|
| Location address | `updateLocation(deviceId, address)` |
| Location coordinates | `updateLocationCoordinates(deviceId, latitude, longitude)` |
| Master valve | `updateControllerMasterValve(zoneNumber, controllerId)` |
| Program mode (STANDARD/ADVANCED) | `updateControllerProgramMode(controllerId, programMode)` |
| Hibernate / wake | `hibernateController` / `wakeController` |
| Expanders | `createExpander` / `updateExpander` / `deleteExpander` |
| Controller notes | `createControllerNote` / `updateControllerNote` / `deleteControllerNote` |
| Zone notes | `createZoneNote` / `updateZoneNote` / `deleteZoneNote` |
| Per-zone master valve | already covered by `updateZoneAdvanced` (we don't capture it today, but the existing mutation accepts it implicitly via masterValve field — needs verification) |
| Run-time groups | **no standalone CRUD** — managed inside `createStandardProgram` / `updateStandardProgram` via `zoneRunTimes[]`. Capture for restore reference, not for direct mutation. |

Reusable schedules (referenced by `schedule_adjustment_ids`) have no CRUD mutation — they appear to be Hydrawise-managed defaults. We capture the IDs in use; if a snapshot references an ID that no longer exists, restore will fail and the user must reconcile manually.

## Goals / Non-Goals

**Goals:**
- Snapshot is self-sufficient for STANDARD-mode restore: every read field has a write tool path, or is documented as "manual restore only" with a `_caveat`.
- Snapshot inlines the StandardProgram details so a restore doesn't need to call additional tools to reconstruct the schedule.
- New write tools for the controller-config category (location, master valve, program mode, expanders) and notes category, following existing project conventions (`runTool`, `previewOrApply`, `PHYSICAL ACTION:`).
- The "writable but not readable" zone fields (watering_mode, etc.) get explicit `_unreadable_fields` markers in the snapshot output so the AI knows what's missing during restore.

**Non-Goals:**
- Sensor capture or sensor restore — Phase 2.
- Advanced-mode programs — Phase 3 (per-zone advanced settings + AdvancedProgram reads).
- Restore orchestration / skill / `_restore_recipe` block — Phase 4.
- Run-time group CRUD as standalone tools — they're scoped inside Standard program mutations and don't have a separate mutation path.
- Reusable schedule CRUD — not exposed by the API.

## Decisions

### Snapshot format is mutable; no migration ceremony

The `snapshot_version` field is retained as informational provenance ("this snapshot was generated by version N of the snapshot tool") but is not a wire-format contract. AI consumers tolerate any shape. This phase bumps the marker to `2`. Future phases bump it again per change without ceremony.

**Alternative considered:** keep v1, add fields additively. Rejected because the inlined program detail under `controller.programs[]` is a substantively different shape (each program now carries its own schedule, not just id/name) — adding it under v1 would mislead readers.

### Inline StandardProgram details inside the snapshot, not separately

`dump_controller_snapshot` calls `getStandardProgram` for each program returned by `list_programs` and replaces the existing thin entry with the full detail. This costs one round-trip per program (typically 1-3 programs per controller); negligible vs. the existing per-zone fan-out.

**Alternative considered:** keep the snapshot thin and require restore to call `get_program` per program separately. Rejected because it splits the backup across two tools and an AI orchestrating restore has to know to do that — the snapshot should be self-contained.

### `_unreadable_fields` block per zone

The snapshot includes a small `_unreadable_fields` array per zone listing the writable-but-not-readable field names: `watering_mode`, `global_master_valve`, `schedule_adjustment_ids`, `watering_type`, `run_time`, `watering_frequency_mode`, `fixed_watering_frequency`, `smart_watering_frequency`, `virtual_solar_sync_watering_frequency`, `run_next_available_start_time`, `pre_configured_watering_schedule_id`, `factors`, `sensor_ids`, `reusable_schedule`, `reusable_schedule_name`, `flow_monitoring_method`, `current_monitoring_method`, `flow_monitoring_value`, `current_monitoring_value`. The AI restoring uses this list to know which fields cannot be inferred from this snapshot alone and must be supplied from another source (a previous snapshot, user input, defaults).

**Alternative considered:** stash these fields as `null` and let the AI infer (already happens in the current serializer). Rejected because the absence of a field name is hard to spot; an explicit list is unambiguous.

### Capability split: `irrigation-controller-config` and `irrigation-notes` as new capabilities

Rather than dumping new tools into existing files, group by domain:

- `irrigation-controller-config` — `update_location`, `update_controller_master_valve`, `update_controller_program_mode`, `hibernate_controller`, `wake_controller`, expander CRUD. Tool registration in a new `src/tools/controllerConfig.ts`.
- `irrigation-notes` — note CRUD for both controller and zone scopes. Tool registration in a new `src/tools/notes.ts`.

This keeps the existing files focused (scheduling.ts is already 600+ lines) and lets these capabilities evolve independently.

### `update_location` is two mutations under one tool

`updateLocation(address)` and `updateLocationCoordinates(lat, lng)` are separate upstream mutations. The `update_location` tool accepts an optional `address` and optional `latitude`/`longitude` and dispatches the relevant mutation(s). At least one must be provided. Preview mode shows what each underlying call would send.

**Alternative considered:** two separate tools (`update_location_address` and `update_location_coordinates`). Rejected because for restore the AI almost always wants to set both at once, and the GUI presents them as one screen.

### Standard mode is the only target this phase

This phase ships restore-completeness for STANDARD-mode controllers only. ADVANCED-mode capture and restore is Phase 3. The new `update_controller_program_mode` tool is included here because it's prerequisite for Phase 3 (you need to be able to switch modes during restore) and the mutation is trivial.

## Risks / Trade-offs

- **Round-trip cost on snapshot**: adding `getStandardProgram` per program adds N additional GraphQL round-trips. For typical accounts (1-3 programs) this is negligible; for accounts with many programs it could matter. Mitigation: parallelize the calls via `Promise.all` (already the pattern in `dump_controller_snapshot`).
- **`Zone.masterValve` semantics**: the schema declares it as `Int!` with documentation "-1 means global, 0 means always disabled, else specific zone". The `updateZoneAdvanced` mutation does not appear to expose this argument explicitly — needs verification. If not writable per-zone, the snapshot captures it for inspection only and we document it as such.
- **Reusable schedules can't be restored if missing**: if a snapshot references a `schedule_adjustment_id` that no longer exists in the account, restore will fail at the `update_zone_settings` call. We surface this in the snapshot with a comment noting external dependency.
- **Run-time groups aren't standalone-restorable**: they're created/updated implicitly via `createStandardProgram` / `updateStandardProgram` via the `zoneRunTimes[]` argument. The snapshot captures the run-time group id+duration alongside the program for inspection, but restore is achieved by replaying the program mutation.
